# The Design Pack — Output Contract (Discovery Studio → Claude Design)

> **What this is.** The exact thing your Discovery Studio *produces* at the end of a session, and the exact thing **Claude Design** *consumes* to design the design system, brand guidelines, logo, and website/app screens. It is the hand-off contract between the two — define it once, and the rest of the build (schema, app) is shaped to produce it.
>
> **One rule that makes everything work:** the **design tokens are the single source of truth.** Claude Design *designs against* them; Claude Code later *exports them to* code. Define them here, in the pack, and the design and the build can never drift apart.
>
> **Format:** a single folder (zipped on export) that a person can read and Claude Design can ingest in one drop.

---

## 1. How it's used (the 30-second flow)

1. You finish a discovery session in **Discovery Studio**.
2. Studio generates **one `design-pack/` folder** (and a `.zip`) — automatically, from the structured answers.
3. You open **Claude Design**, drop in the pack, and point it at `README.md`.
4. Claude Design reads the brief + tokens + references and designs the requested deliverables.
5. High-fi designs → **Claude Code** → GitHub → Vercel + Supabase.

The pack is **self-contained and self-explanatory** — no memory of the session required.

---

## 2. Folder structure

```
design-pack/
├── README.md                  # index + how Claude Design should use this pack
├── brief.md                   # the narrative design brief (human-readable)
├── brand-data.json            # the typed Brand Data Object (machine source of truth)
│
├── tokens/
│   └── brand.tokens.json      # DTCG 2025.10 design tokens (color, type, spacing, radius, shadow)
│
├── references/
│   ├── references.json        # manifest: every reference + sentiment (love / like / avoid)
│   ├── inspiration-01.jpg     # uploaded inspiration / mood images
│   ├── competitor-01.png      # competitor screenshots
│   └── ...
│
├── assets/                    # client-OWNED existing assets (only if they have them)
│   ├── logo-current.svg
│   ├── brand-guidelines-old.pdf
│   └── product-photo-01.jpg
│
├── deliverables/              # ONE brief per thing Claude Design must design
│   ├── design-system.md
│   ├── brand-guidelines.md
│   ├── logo.md
│   ├── website.md             # includes the screen + component list
│   ├── social.md              # (optional)
│   └── packaging.md           # (optional, product brands)
│
└── content/                   # real copy so screens aren't lorem ipsum (optional)
    └── website-copy.md
```

> **Minimum viable pack** (required every time): `README.md`, `brief.md`, `brand-data.json`, `tokens/brand.tokens.json`, `references/`. Everything else is included when the session captured it.

## 3. File-by-file spec

### 3.1 `README.md` — the index + instructions for Claude Design
Top of the pack. Two jobs: list what's inside, and **tell Claude Design how to use it.** Ends with an explicit instruction block.

```md
# {Client} — Design Pack
Generated by Discovery Studio · {date} · project {id}

## What's inside
- brief.md — the design direction in plain language
- brand-data.json — structured source of truth (colors, type, logo, audience…)
- tokens/brand.tokens.json — design tokens (DTCG). DESIGN AGAINST THESE.
- references/ — inspiration + competitors, tagged love/like/avoid
- assets/ — client's existing logo/guidelines (if any)
- deliverables/ — one brief per thing to design

## Instructions for Claude Design
1. Read brief.md for direction, then brand-data.json for exact values.
2. Use tokens/brand.tokens.json as the source of truth for color, type, spacing — do not invent new values; extend the scale if needed and note it.
3. Study references/ — match the "love" set, avoid the "avoid" set.
4. Design each item in deliverables/ in its listed order.
5. Flag any missing input instead of guessing; keep everything on-token.
```

### 3.2 `brief.md` — the narrative brief
The plain-language story a designer reads first. Sections:
- **Snapshot** — who they are, product vs service, what they sell (2–3 lines).
- **Audience** — who it's for, in human terms.
- **Brand personality & archetype** — e.g. "Creator archetype; bold, modern, minimal" + the captured mood words.
- **Voice & tone** — formal↔casual, we/you, emoji or not, with 1–2 example sentences.
- **Visual direction** — the style cluster (e.g. "techy-futuristic, dark-first"), with the *why*.
- **Logo direction** — which of the 7 types they chose and why.
- **Color direction** — emotional direction + locked colors (plain words; exact values live in tokens).
- **Must-haves & no-gos** — explicit do/don't.

> Written in plain English, not jargon — it should read like a smart creative brief.

### 3.3 `brand-data.json` — the typed source of truth
The full **Brand Data Object** (schema already defined in the blueprint, §9). This is the machine-readable backbone — every field the session captured: `business`, `audience`, `goals`, `brand`, `voice`, `logo`, `color`, `type`, `visualStyle`, `imagery`, `surfaces`, `product`, `references`, `constraints`, `meta`. Claude Design reads exact values from here; `brief.md` is the readable narration of the same data.

### 3.4 `tokens/brand.tokens.json` — DTCG design tokens
The contract. Conforms to **DTCG 2025.10** (`$type` / `$value`). Colors include brand + neutrals + semantics; plus type, spacing, radius, shadow. Example:

```json
{
  "color": {
    "brand": {
      "primary":   { "$type": "color", "$value": "#C6FF3A", "$description": "Electric lime — primary accent" },
      "secondary": { "$type": "color", "$value": "#00E5FF" }
    },
    "neutral": {
      "0":   { "$type": "color", "$value": "#0A0A0B" },
      "900": { "$type": "color", "$value": "#FFFFFF" }
    },
    "semantic": {
      "success": { "$type": "color", "$value": "#22C55E" },
      "danger":  { "$type": "color", "$value": "#EF4444" },
      "warning": { "$type": "color", "$value": "#F59E0B" },
      "info":    { "$type": "color", "$value": "#3B82F6" }
    }
  },
  "font": {
    "family": {
      "display": { "$type": "fontFamily", "$value": ["Clash Display", "sans-serif"] },
      "body":    { "$type": "fontFamily", "$value": ["Inter", "sans-serif"] },
      "mono":    { "$type": "fontFamily", "$value": ["JetBrains Mono", "monospace"] }
    },
    "scale": {
      "display": { "$type": "dimension", "$value": { "value": 72, "unit": "px" } },
      "h1":      { "$type": "dimension", "$value": { "value": 48, "unit": "px" } },
      "body":    { "$type": "dimension", "$value": { "value": 16, "unit": "px" } }
    }
  },
  "spacing": {
    "1": { "$type": "dimension", "$value": { "value": 4,  "unit": "px" } },
    "2": { "$type": "dimension", "$value": { "value": 8,  "unit": "px" } },
    "4": { "$type": "dimension", "$value": { "value": 16, "unit": "px" } }
  },
  "radius": {
    "sm":   { "$type": "dimension", "$value": { "value": 4,  "unit": "px" } },
    "lg":   { "$type": "dimension", "$value": { "value": 16, "unit": "px" } },
    "full": { "$type": "dimension", "$value": { "value": 9999, "unit": "px" } }
  },
  "shadow": {
    "card": { "$type": "shadow", "$value": {
      "color": "#00000022", "offsetX": {"value":0,"unit":"px"}, "offsetY": {"value":8,"unit":"px"},
      "blur": {"value":24,"unit":"px"}, "spread": {"value":0,"unit":"px"} } }
  }
}
```

> Discovery captures the *picks* (locked colors, color direction, type feel); the export step **derives** the full ramp — neutrals, semantics, type scale, spacing — so Claude Design gets a complete, usable token set, not just two hex codes. File uses media type `application/design-tokens+json`, extension `.tokens.json`.

### 3.5 `references/` — the visual evidence (highest-signal input)
- Image files dropped straight in (inspiration, competitor screenshots, client uploads).
- `references.json` manifest tags each one:

```json
[
  { "file": "inspiration-01.jpg", "type": "inspiration", "source": "upload", "sentiment": "love",
    "note": "loves the bold type + dark background", "extracted": { "palette": ["#0A0A0B","#C6FF3A"], "fonts": ["grotesk"] } },
  { "file": "competitor-01.png", "type": "competitor", "source": "https://competitor.com", "sentiment": "avoid",
    "note": "too corporate / generic" }
]
```

> `sentiment` (love / like / avoid) is what makes references actionable — Claude Design matches the "love" set and steers clear of "avoid."

### 3.6 `assets/` — the client's existing materials
Only if they have them: current logo (prefer SVG), old guidelines PDF, product photos, fonts they own. Copied in as-is so Claude Design can reuse or evolve them.

### 3.7 `deliverables/*.md` — one brief per thing to design
Each file = a focused brief for one output, so Claude Design designs them in order. Each contains: **goal, scope, specific requirements, what to use from the pack, and acceptance criteria.** The important ones:

- **`design-system.md`** — the full token set + components to produce (buttons, inputs, cards, badges, etc.) and the preview structure. (This is your VOLT brief, now living inside the pack.)
- **`brand-guidelines.md`** — assemble the 6 standard sections: logo usage, color, typography, voice, imagery, plus do/don't.
- **`logo.md`** — chosen logo type(s), must-convey ideas, variations needed (primary/stacked/mark/mono), color-on-light/dark.
- **`website.md`** — **the screen list + component list per surface**, layout feel, and which content file to use. Example:
  ```md
  ## Surfaces & screens
  - Marketing site: Home, Work, Project detail, About, Services, Contact
  - Components: header, slider, project card, meta grid, footer, CTA band
  Layout feel: dark-first, fullscreen slider, big type, lively motion.
  Use content/website-copy.md for real copy.
  ```
- **`social.md`, `packaging.md`** — included only when the session scoped them.

### 3.8 `content/*.md` — real copy
Any copy captured (taglines, about text, product blurbs) so designs render with real words. Optional but valuable.

## 4. Conventions
- **Encoding/format:** UTF-8; Markdown for human docs, JSON for machine files.
- **Tokens:** DTCG 2025.10, `.tokens.json`, `$type`/`$value`. No proprietary formats.
- **Images:** keep originals; reference them by relative path; never inline base64 in JSON.
- **Naming:** lowercase, hyphenated, numbered (`inspiration-01.jpg`).
- **Determinism:** the same answers always produce the same pack structure — so Claude Design always knows where to look.
- **Self-contained:** every path is relative; the zip works on any machine.

---

## 5. Why this design
- **Brief + data split** — `brief.md` (prose) and `brand-data.json` (typed) describe the same thing two ways, so both a human and Claude Design are well served.
- **Tokens as contract** — the one file that travels unchanged from Design to Code; prevents drift.
- **References as first-class** — the highest-signal design input gets its own folder + sentiment tags.
- **One brief per deliverable** — Claude Design works through a clear, ordered to-do instead of one giant prompt.
- **Minimum-viable core** — even a quick session yields a valid, useful pack.

---

## 6. What Discovery Studio must do to produce this (feeds the Phase 1 spec)
1. Store every answer as **structured fields** (the Brand Data Object), not free text.
2. On export, **derive** the full token ramp (neutrals/semantics/scales) from the picks.
3. Bundle uploaded references + assets and write the manifests.
4. Generate `brief.md` and each `deliverables/*.md` from the structured data via the LLM.
5. Zip `design-pack/` and offer download + (optionally) push to a shared location.

That list *is* the seed of the Phase 1 build spec — the schema and app exist to emit this pack.
